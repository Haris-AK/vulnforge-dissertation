# Evaluation patterns for CM3203 projects

Use this file when evaluation advice depends on likely project type. Do not force a pattern if the project clearly needs a different one.

## How to use this file

1. Infer the likely project type from the student's draft or question.
2. Suggest evaluation patterns that match that type.
3. Tie the advice back to the student's aims, objectives, deliverables, or claims.
4. Prefer a few realistic evaluation moves over an ideal but infeasible plan.

## Software or systems project

Common evaluation dimensions:
- functional correctness
- performance or efficiency
- robustness or reliability
- usability, where relevant
- comparison with stated requirements or baseline approaches

Helpful advice:
- ask what the system is supposed to do, then what evidence would show it does that well
- encourage test cases tied to main features and failure conditions
- encourage at least one form of non-trivial evidence beyond "it runs"

## Machine learning or data project

Common evaluation dimensions:
- metric choice and justification
- baselines or comparison points
- train/validation/test split or equivalent methodology
- error analysis or qualitative inspection
- ablations or sensitivity checks where realistic

Helpful advice:
- encourage the student to justify metrics rather than just report them
- ask whether there is a simple baseline that would make the results interpretable
- highlight risks around data leakage, overfitting, and unexamined failure cases

## Visualisation, HCI, or UI project

Common evaluation dimensions:
- task-based effectiveness
- usability
- comparison to alternatives or prior design
- readability, clarity, or discoverability
- user feedback where appropriate and feasible

Helpful advice:
- encourage evaluation against realistic user tasks
- if a user study is not feasible, suggest lighter-weight structured evaluation rather than no evaluation
- ask what the visualisation or interface helps users do better

## Theoretical, mathematical, or conceptual project

Common evaluation dimensions:
- correctness of reasoning
- clarity of assumptions
- illustrative examples or worked cases
- limitations and boundary conditions
- comparison with alternative formulations where appropriate

Helpful advice:
- encourage explicit assumptions and scope limits
- use examples to show what the method or argument achieves
- ask what would count as convincing evidence for the main claim

## Hardware, embedded, or systems integration project

Common evaluation dimensions:
- measurements under realistic conditions
- constraints and trade-offs
- reliability and repeatability
- integration success
- comparison with requirements or expected tolerances

Helpful advice:
- encourage reporting conditions, setup, and limits clearly
- distinguish design intent from measured behaviour
- highlight safety, reliability, or reproducibility issues where relevant

## Incomplete project

If the project is incomplete, evaluation can still be useful through:
- partial testing of finished components
- honest analysis of unmet aims or blocked deliverables
- evidence of what was achieved and what remains uncertain
- a clear explanation of what would be needed next

Avoid implying that lack of completion prevents meaningful evaluation.
