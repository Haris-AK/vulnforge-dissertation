# Student help topics for CM3203

Use this file as a practical playbook for common student questions.

## Initial plan

Helpful advice:
- The initial plan is a short planning document (main body max 1,500 words), not the full report.
- The structure should follow the guide closely: project description, aims and objectives, feasibility, work plan.
- The work plan should cover the required period (e.g. spring weeks 1–12), including initial plan preparation.
- Milestones are things that are provably achieved; tasks lead to milestones; deliverables are the resulting outputs.
- A weekly scale is usually a good default unless another format is clearly better.
- Encourage a **specific** project title and description (exact problem and intended output), not a generic topic area.
- Encourage 2–4 concrete objectives rather than a long wish-list.
- Aims should be clear; objectives should be concrete, actionable, and linked to later deliverables and evaluation.
- If scope is too broad, narrow to a feasible core contribution with a realistic evaluation path.
- Keep the plan concise and scannable; avoid vague prose.

## Initial plan common failure modes

Watch for:
- a title that names only a topic area rather than a project
- a project description that explains background but not the exact problem or intended output
- objectives that are broad intentions rather than specific actions
- objectives with no clear route to validation or evaluation
- a work plan that lists activity but not milestones or deliverables
- feasibility or risks listed without mitigation
- an over-ambitious scope for one semester
- a work plan that is technically present but visually unreadable

## Report structure and argument

Helpful advice:
- The reader should understand the exact problem, why it matters, why the chosen approach makes sense, what was built or done, and how success was evaluated.
- If a chapter feels weak, the common fix is not more volume but a clearer logical bridge between: problem -> approach -> artefact -> evidence -> conclusions.
- Encourage explicit aims/objectives or research questions when the draft is vague.
- Encourage students to make their **own contribution** visible, especially when they used existing frameworks, libraries, datasets, or tools.

## Methods, implementation, and tools

Helpful advice:
- Students should justify important technical choices, not just describe them.
- If the project uses AI or automation tools materially, students should explain what they used, why, and how they validated outputs.
- A strong methods/implementation section usually explains trade-offs and rejected alternatives where relevant.

## Evaluation and evidence

Helpful advice:
- Students often under-specify evaluation criteria. Encourage criteria tied to aims or deliverables.
- Results should be interpreted, not just listed.
- Limitations, failure cases, and unmet deliverables should be analysed directly.
- Where a system is incomplete, students can still write a good report if they are honest, evidence-based, and evaluative.
- Use `evaluation-patterns.md` when evaluation expectations depend strongly on project type.

## Reflection and project management

Helpful advice:
- Reflection should analyse decisions, changes, trade-offs, lessons learned, and what the student would now do differently.
- Project management discussion should explain what changed over time, not merely narrate a timeline.

## Submission and evidence selection

Helpful advice:
- Encourage students to submit the report plus the key artefacts needed to evidence the work.
- Students do not need to dump every raw file if it does not help assessment.
- For support files, a curated, explained selection is usually more helpful than an unstructured archive.

## Project changes and scope control

Helpful advice:
- It is normal for projects to evolve.
- The important issue is whether changes were understood, justified, and agreed where necessary.
- When a student is over-scoped, help them narrow to a feasible core problem with an evaluable contribution.

## Student-facing phrasing pattern

When answering questions, it often helps to use this sequence:
1. state the likely guide-based answer
2. explain what it means in practice
3. give the smallest next action that would improve the student's position

## Workload-reduction principle

Prefer answers that would genuinely save a supervisor time:
- answer the question directly
- convert vague confusion into a concrete next step
- point out the one or two most important report or project issues first
- avoid overwhelming students with every possible improvement at once
