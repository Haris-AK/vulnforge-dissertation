# Chapter 4 Implementation

## 4.1 Introduction

This chapter describes the implementation of VulnForge, the AI-assisted cybersecurity training platform developed for this project. It is written with the initial plan as the organising reference point, particularly the four objectives defined there: an orchestration controller for ephemeral environments, a generative vulnerability engine, a web dashboard with SSH-based access and flag-based completion, and a context-aware LLM tutor. The implementation translates those planned objectives into a working system that can provision disposable Linux containers, inject intentionally vulnerable configurations, expose those containers to a learner through a controlled network path, verify challenge completion through a flag mechanism, and provide guided assistance through an integrated large language model (LLM) tutor. Rather than treating the platform as a single monolithic application, the implementation was organised as a set of cooperating subsystems: infrastructure orchestration, vulnerability generation, challenge lifecycle management, persistence and authentication, the user-facing dashboard, and the tutor component.

The problem framing established earlier in the project was that many existing cybersecurity training platforms provide either static challenges or limited support when a learner becomes stuck. This directly reflects the initial plan's argument that platforms such as Hack The Box and TryHackMe can feel static and offer limited personalised help once a student is blocked. A major design goal for VulnForge was therefore to combine the realism of hands-on exploitation with the flexibility of generative AI. In practical terms, this meant that the implementation had to solve two problems at once. First, it had to create and manage isolated environments reliably enough for repeated student use. Second, it had to constrain a generative model so that its output was not just plausible prose, but a structured set of deployment instructions that could be validated and executed safely.

The final implementation differs in several useful ways from the earliest prototype ideas set out in the plan. The plan left open whether the application layer would be built in a more script-led style and whether the LLM would be hosted locally or in the cloud depending on resource constraints. In the implemented system, however, the application layer was built as a TypeScript full-stack web application using Next.js, while the infrastructure automation was retained in Ansible and the generative component was implemented through the Google Gemini API. Access was also realised through Tailscale-routed SSH to isolated LXC containers on Proxmox rather than through a broader, less bounded networking model. These changes did not alter the project aim; they were implementation decisions taken to satisfy the same core objectives more robustly within the available hardware, time and safety constraints. This architecture was chosen because it allowed the project to separate concerns clearly: the web application handles user interaction, validation, persistence and orchestration; the LLM produces structured vulnerability specifications; and Ansible interacts with Proxmox and the containers. This separation made the system easier to debug and allowed each layer to use tools that fit its role.

The chapter is structured around the main implementation decisions. Section 4.2 presents the overall system architecture and explains how requests move through the platform. Section 4.3 discusses the infrastructure layer, including Proxmox, LXC containers, Ansible orchestration, network access, and automatic cleanup. Section 4.4 explains the vulnerability generation pipeline, focusing on the schema-driven use of Gemini, prompt design, diversity controls, and the injection workflow. Section 4.5 then describes the web application layer, covering authentication, persistence, repository design, and challenge lifecycle APIs. Section 4.6 examines the frontend dashboard and supporting interfaces, while Section 4.7 covers the tutor subsystem and its safeguards. Section 4.8 reflects on security, robustness, and practical trade-offs that emerged during implementation. Taken together, these sections show how the project objectives were converted into a coherent end-to-end system rather than a disconnected collection of prototypes.

## 4.2 Overall system architecture

VulnForge was implemented as a layered system in which the web application acts as the single entry point for both the learner and the infrastructure pipeline. At the highest level, a student authenticates through the web interface, selects a vulnerability category and difficulty, and launches a challenge. That action triggers a server-side orchestration process. The application generates a structured vulnerability specification through Gemini, provisions a fresh LXC container on a Proxmox host through Ansible, injects the vulnerable artefacts into the container, stores challenge metadata in a local database, and returns the connection details required for the learner to begin interacting with the environment. While the challenge is active, the same application provides flag submission, access to the AI tutor, account settings, and a history view of completed challenges.

This architecture deliberately places the Next.js application at the centre of the system rather than exposing infrastructure automation directly to the client. There are several reasons for this. First, it keeps all security-sensitive operations server-side. The learner never interacts directly with Proxmox, Ansible, or the LLM API, which reduces the attack surface and makes it easier to enforce application-level policies. Second, it creates a natural point at which user input can be validated before it affects the infrastructure. For example, category selection, difficulty selection, SSH key submission, chat messages, and flag submissions are all validated in API routes before any state changes occur. Third, it allows the platform to keep the database as the authoritative record of challenge ownership, status, expiry and tutor context, which is essential for maintaining consistency between the visible user interface and the actual infrastructure state.

From an implementation perspective, the system is divided into four principal technical layers. The first is the application layer, built with Next.js 16, React 19 and TypeScript. This layer contains both the frontend pages and the backend route handlers. The second is the persistence layer, implemented with SQLite, better-sqlite3 and Drizzle ORM, which stores users, sessions, challenge records and tutor messages. The third is the generation layer, implemented through the Google Gemini SDK and a custom prompt-and-schema pipeline. The fourth is the infrastructure layer, built around Ansible playbooks and a Proxmox host running unprivileged Ubuntu 24.04 LXC containers. These layers communicate through explicit contracts: JSON request schemas between the frontend and API, Zod-derived types between the LLM and the application, and Ansible extra variables between the application and the infrastructure automation.

Viewed directly against the initial plan, the mapping from proposal to implementation is straightforward. Section 4.3 addresses Objective 1 by showing how the orchestration controller provisions containers from a reusable template, configures network access, and injects the learner's SSH key. Section 4.4 addresses Objective 2 by explaining how the LLM generates a bounded vulnerability specification that can then be applied to the container. Sections 4.5 and 4.6 address Objective 3 through the dashboard, account management, challenge launch flow, SSH-facing access model and flag-based completion process. Section 4.7 addresses Objective 4 through the challenge-aware tutor and its anti-leak safeguards. The initial plan's wider feasibility concerns, such as time-limited environments, safe isolation and practical resource constraints, are reflected throughout Sections 4.3 and 4.8.

A key architectural choice was to treat the LLM as a component in a controlled pipeline rather than as a free-form code generator. The model is never allowed to emit arbitrary text that is then executed directly. Instead, Gemini is instructed to return a JSON object that conforms to a vulnerability schema. That schema describes a bounded set of deployment primitives: packages to install, files to create, services to restart or enable, setup commands to run, descriptive metadata for the learner, and an internal solution summary for tutoring. The application validates this response before passing it to Ansible. This design is central to the project because it moves the LLM from an unstructured assistant role into a constrained specification generator. As a result, the system can reject malformed outputs before they reach the container environment.

Another important architectural feature is the split between logical challenge state and physical container state. The database records whether a challenge is active, completed, destroyed or expired, while the infrastructure layer determines whether a given container is running on the Proxmox host. In an ideal system these states always align, but real systems inevitably encounter race conditions, failures and delayed cleanup. The implementation therefore uses multiple mechanisms to keep them aligned: explicit destroy operations, challenge expiry checks in the API, a one-active-challenge constraint in the database, and a host-level TTL reaper that destroys long-lived containers even if the application is unavailable. This is a good example of the project’s general implementation philosophy: instead of assuming that each subsystem will behave perfectly, the design uses overlapping safeguards so that failures in one layer do not immediately break the entire platform.

## 4.3 Infrastructure orchestration and ephemeral lab provisioning

### 4.3.1 Proxmox and the container model

The infrastructure objective in the initial plan was to create a controller capable of provisioning isolated, disposable machines for cybersecurity exercises. In the final system, this was implemented using Proxmox VE as the bare-metal hypervisor and unprivileged Ubuntu 24.04 LXC containers as the actual challenge environments. Containers were chosen over full virtual machines because they provided a more efficient use of limited hardware resources while still supporting realistic Linux-based exploitation scenarios. For the scope of this project, LXC containers offered an effective compromise: they are lightweight enough to provision quickly, but still expose a full user space, filesystem hierarchy, systemd services, package management, and network interfaces.

The provisioning logic assumes the existence of a reusable Ubuntu template on the Proxmox host. When a challenge is launched, Ansible clones from this template rather than constructing each environment from scratch. The deploy playbook specifies defaults such as CPU, memory, disk size, bridge configuration, DNS settings and network setup. These defaults are intentionally modest, both because the platform is designed for one-semester project scope and because each container is intended to exist only long enough for the learner to attempt the exercise. The chosen defaults also support an educational workload more effectively than trying to emulate a large production environment. In other words, the platform prioritises repeatable challenge provisioning over heavyweight realism.

The containers are created as unprivileged LXC instances. This is an important design choice from a safety perspective, because it narrows the range of operations a compromised container could perform against the host. This aligns with the project’s requirement that learners interact only with the intentionally vulnerable environment rather than with the hypervisor itself. It also shaped later prompt constraints, because the LLM could not be allowed to generate challenge content that assumed privileged kernel features, host-level access, or container capabilities unavailable in unprivileged mode.

### 4.3.2 Ansible as the orchestration layer

Although the application layer was ultimately implemented in TypeScript, infrastructure automation was intentionally kept in Ansible. The reason was not simply convenience. Ansible provided a well-defined, repeatable, and auditable way to express container lifecycle operations and post-provisioning steps. Instead of embedding hypervisor logic directly into application code, the web application invokes playbooks as child processes using Node’s execFile API. This gives the application a clean way to pass structured variables to the infrastructure layer while still allowing Ansible to handle SSH, task sequencing, retries and Proxmox-specific modules.

The deploy and destroy workflows are implemented through a reusable role, `lxc_lifecycle`, which supports both creation and destruction depending on the state variable passed to it. During deployment, the playbook downloads the Ubuntu template if necessary, creates the LXC container through the `community.proxmox.proxmox` module, starts it, creates the non-root student user, injects the learner’s SSH public key, waits for the container to receive a DHCP address, extracts the assigned IP, and writes an artefact file describing the deployment. During destruction, the same role removes the specified VMID and records a destroy artefact. This reuse is valuable because it keeps lifecycle logic in a single place rather than duplicating it across scripts.

One subtle but important implementation detail is how SSH key injection is handled. An early, insecure approach would have interpolated the public key directly into a shell command. The final implementation instead writes the key to a temporary file on the host, uses `pct push` to transfer it into the container, and then sets ownership and permissions appropriately. This decision is a good example of the project’s move from “make it work” automation to safer automation. User-controlled content is never placed unescaped into host-side shell commands, which reduces the risk of accidental command injection and makes the system’s trust boundary clearer.

The application layer wraps these playbooks in `app/lib/ansible-runner.ts`. This wrapper resolves playbook and inventory paths, constructs argument lists, sets the Ansible JSON callback, imposes timeouts, and parses structured output. The use of `execFile` rather than `exec` is deliberate. Because arguments are passed as a list rather than as a shell string, the application avoids a class of shell injection risks that would arise if extra variables were concatenated into a command line. The runner also stages vulnerability specifications in a temporary directory as JSON files, allowing the inject playbook to consume them without complex quoting.

### 4.3.3 Network access and learner connectivity

A challenge platform is only useful if learners can reach the environments it provisions. In VulnForge, network access is provided through Tailscale subnet routing rather than by exposing containers directly to the public internet. The Proxmox host advertises the internal container subnet to authorised clients. This gives the learner a relatively straightforward way to SSH to the challenge using the assigned IP address, while keeping the actual container network under the control of the host. In practice, once a challenge is provisioned the dashboard can display a command of the form `ssh student@<container-ip>`, and the learner connects using the SSH key they previously uploaded in account settings.

This approach was selected because it balanced accessibility with containment. Opening challenge services directly to the internet would have been simpler from the learner’s perspective, but it would have created a much weaker security posture for a platform whose entire purpose is to run deliberately vulnerable systems. Tailscale provides encrypted transport and access control at the network layer, while the platform still enforces per-container identity through the injected SSH key. Each container only trusts the key submitted by the user who launched it, meaning that even if multiple learners can route to the same subnet, they cannot simply log in to another learner’s machine without the appropriate credentials.

There are practical trade-offs in this design. The network path is more secure than public exposure, but it introduces onboarding requirements and some complexity around client setup. The repository includes analysis of alternative networking approaches, which reflects that this remained an active design consideration. For the implemented version, however, the key point is that the network layer is treated as part of the security model rather than just as plumbing. Challenge access is intentionally mediated rather than left open by default.

### 4.3.4 Automatic cleanup and the TTL reaper

Because challenges are intended to be ephemeral, the platform requires reliable cleanup. Simply relying on the learner to destroy a container after use would be inadequate, especially on limited hardware. The application therefore uses a one-hour time-to-live for active challenges, recorded as `expires_at` in the database, and supplements this with a host-level cleanup script. The database layer is responsible for user-visible state transitions such as active, completed, destroyed and expired. However, the infrastructure layer includes a cron-driven TTL reaper on the Proxmox host that scans running containers tagged as belonging to VulnForge, checks their uptime, and destroys any that exceed the configured lifetime.

This two-layer cleanup strategy is an example of defence in depth. If the web application remains healthy, expiry is detected through challenge APIs and stale challenges are marked accordingly. If the application fails, the browser is closed, or the database state is not updated promptly, the host still reclaims resources. The TTL reaper script performs graceful shutdown first, then escalates to force stop and destroy if needed, and logs its actions. This is important because container cleanup is not just a convenience feature. It is part of the project’s answer to the risk, identified in the initial plan, that vulnerable environments must not remain accessible indefinitely.

## 4.4 LLM-driven vulnerability generation and injection

### 4.4.1 Structured generation rather than free-form generation

The vulnerability generation subsystem is the most distinctive part of VulnForge. The goal was not merely to use an LLM to write challenge descriptions, but to use it to generate deployable vulnerability specifications that could produce functioning and exploitable lab environments. In practice, this required the project to solve a common problem with generative systems: model outputs can be plausible yet operationally useless. The implementation addresses this by placing Gemini behind a schema-enforced generation pipeline.

The schema is defined in `app/lib/schemas/vulnerability.ts` using Zod. It specifies the exact structure expected from the model, including the title, description, category, difficulty, hint, required apt packages, files to write into the container, services to manage, optional setup commands, an internal solution summary, and an attack vector label. This schema acts as a contract between the model and the rest of the system. The application converts the schema to JSON Schema for Gemini’s structured output mode and then validates the returned JSON again with Zod before any infrastructure operation occurs. This double validation is important because it means the LLM cannot bypass the project’s expected structure through prompt-compliant but malformed output.

The design of the schema reflects lessons learned during implementation. For example, the `files` array must contain at least one entry so that the platform does not accept an empty challenge definition. File permission modes are constrained to octal-like strings, and service actions are restricted to a small set that the playbook knows how to implement. Optional commands are explicitly separated into `setup_commands` instead of allowing the model to blur together configuration and execution. This is valuable because it limits the operational vocabulary of the model. The platform is not executing arbitrary “AI-generated code”; it is executing a validated description built from known primitives.

### 4.4.2 Prompt engineering and category design

The project’s initial plan proposed support for different kinds of vulnerability, ranging from web server misconfigurations to logic flaws in web applications. In the implementation, this became a set of explicit categories defined in `app/lib/prompts/categories.ts`: web server misconfiguration, privilege escalation, SQL injection, insecure file permissions, and web application logic flaws. Each category contains a label and description for the frontend, but more importantly it contains a prompt template that tells Gemini how to construct that type of challenge. The category prompt is combined with a system-level prompt that establishes global rules about the environment, allowed packages, the student user model, flag placement, and service management.

The system prompt does a large amount of work. It frames the generation task as educational, constrains the environment to Ubuntu 24.04 unprivileged LXC containers, reminds the model that the learner connects as a non-root `student` user, and specifies that the flag must not be directly readable without exploitation. It also bans common failure modes such as recreating the student user, assuming unavailable services, or relying on host-level access. In effect, the system prompt serves as a specification of the platform’s invariants. This is particularly important because the model is being used to generate intentionally vulnerable content inside a bounded environment. The prompt therefore has to distinguish between acceptable insecurity inside the container and unacceptable behaviour from the platform’s perspective.

The category-level prompts then specialise this behaviour. For example, web application categories force the model to emit complete web server configurations rather than simply assuming that installing a package is enough. This reflects a practical lesson from implementation: many LLM-generated web challenges fail not because the vulnerability is absent, but because the generated service is never actually reachable. The prompt therefore evolved to include highly concrete operational instructions, such as where to place Nginx or Apache configuration files, how to enable sites, and which systemd services to restart. The implementation here illustrates an important point about LLM engineering: successful use of the model required increasingly explicit operational constraints, not simply more “creative” prompting.

### 4.4.3 Diversity controls and repeated challenge generation

A platform that always generates the same canonical version of a category would undermine the project’s aim of producing varied learning experiences. The implementation therefore includes several diversity mechanisms. First, each category defines a set of variants, such as alternative forms of web server misconfiguration or different business logic flaws. At generation time, one variant is selected randomly and injected into the prompt. Second, categories that correspond to web applications also draw from a pool of application themes, such as booking systems, internal portals or management dashboards. This forces variation in the narrative and structure of the generated application, not just in the vulnerability primitive. Third, the system can include a short history of the learner’s previous challenges in the same category, instructing Gemini not to repeat them.

These controls are a meaningful technical contribution because they move the platform away from a one-shot text generation mindset and toward stateful personalised generation. The diversity system is relatively lightweight in code, but it addresses a real educational need. A learner who repeatedly selects “SQL injection” should not receive the same login bypass every time. By varying the variant, application theme and exclusion history, the platform encourages broader exposure within a category and reduces the chance that users simply memorise one exploitation path.

### 4.4.4 Server-side flag generation and model fallback

The current implementation does not allow Gemini to invent its own challenge flag. Instead, the flag is generated server-side using cryptographic random bytes and then inserted into the user prompt as a literal string that must be embedded in the challenge. This design replaced an earlier approach in which the model both generated and reported the flag. That earlier design created several failure modes, including malformed flags, incorrect reported locations, and fragile verification behaviour. By generating the flag server-side, the system decouples challenge correctness from the model’s ability to invent consistent identifiers. The flag becomes an invariant known to the application before generation begins.

The Gemini integration also includes a fallback chain of models. The application constructs an ordered list of models from configuration and attempts generation in sequence when it encounters retryable failures such as overload or rate limiting. This is a small but important robustness feature. Because the project depends on an external API, it cannot assume that every generation request will succeed on the first attempt. The fallback wrapper gives the platform a controlled way to recover from transient LLM-side failures without having to implement complex retry logic throughout the codebase.

### 4.4.5 From specification to running vulnerability

Once the application has a validated vulnerability specification, it passes that specification to the injection pipeline. The implementation here is especially important because it bridges the conceptual gap between “the LLM said to create these files and packages” and “the container now contains a usable vulnerable environment”. The `inject-vulnerability.yml` playbook performs this work in stages. It validates input variables, loads the JSON specification, updates the apt cache inside the container, installs packages, stages generated files on the host, creates parent directories inside the container, pushes the files in with the correct permissions, manages required services, runs optional setup commands, and performs a final service restart.

The use of host-side staging and `pct push` is a deliberate response to the unreliability of shell-based file injection. Gemini-generated file content may contain quotes, backticks, dollar signs, newlines and other characters that make naive shell echo commands unsafe or corrupt. By writing files to a temporary host directory and using Proxmox’s file transfer mechanism, the system avoids a whole class of escaping bugs. This is another example of the project treating LLM output as potentially complex data rather than as a snippet to paste into a shell.

The injection playbook also contains improvements motivated by practical debugging. Setup commands are captured and reported clearly if they fail. Services are restarted again after setup commands, because many application configurations only become active after symlinks or service files are created. Failure messages are extracted from Ansible’s JSON callback output and surfaced to the application so that the frontend can show more useful diagnostics. These changes matter because an LLM-driven system is difficult to develop if every failure simply appears as “generation failed”. The implementation therefore emphasises error surfacing and post-failure cleanup as much as it emphasises successful deployment.

## 4.5 Web application, persistence, and challenge lifecycle management

### 4.5.1 Next.js as a full-stack application layer

The web application was implemented with Next.js using the App Router structure, allowing the project to keep frontend pages and backend route handlers in the same codebase. This was a good fit for the project because the application is not simply a static dashboard layered on top of a separate API. Much of the project’s complexity lies in orchestration logic, validation, and status management, all of which benefit from close coupling between the user interface and server-side behaviour. Using Next.js also simplified deployment of authenticated pages, API routes, and streaming interfaces such as server-sent events for both challenge launch progress and tutor responses.

The application is organised around the `app/` directory. Authentication pages, settings, history pages, and the main dashboard are implemented as page components. API endpoints live alongside them under `app/api`. Shared logic such as authentication helpers, database repositories, Gemini integration, prompt configuration, SSH key validation and Ansible execution resides in `app/lib`. This structure helped keep the project cohesive. Instead of scattering the project across multiple services prematurely, the implementation keeps a single web application responsible for coordination, while still factoring the internal modules cleanly.

### 4.5.2 Authentication and session handling

The initial plan identified the need for a user interface that could manage accounts and SSH keys. In the final implementation, authentication is handled with email/password signup and login, bcrypt-based password hashing, and a session table stored in SQLite. When a user logs in or signs up, the server creates a session record and issues an HTTP-only session cookie. Sessions expire after seven days, and route handlers use a shared `requireAuth()` helper to retrieve and validate the authenticated user. This means that protected actions such as launching a challenge, submitting a flag, fetching active challenge data, viewing history, or updating settings all depend on server-side session validation rather than on client-side trust.

This design is intentionally lightweight. A dissertation project does not require a full enterprise identity stack, and introducing external authentication providers would have added complexity that was not central to the project question. At the same time, the implementation does take basic security seriously: passwords are hashed rather than stored directly, session tokens are placed in HTTP-only cookies, and protected routes are hidden from unauthenticated users through middleware-level checks before the route handlers apply deeper validation.

### 4.5.3 SQLite, Drizzle ORM and repository design

Persistence is implemented through SQLite with better-sqlite3 and Drizzle ORM. This is a sensible choice for the scale of the project. The application is single-instance, educational, and not expected to support large concurrent workloads, so SQLite offers a simpler operational model than introducing a separate database server. However, the project still benefits from a structured data layer, and Drizzle provides typed schema definitions and migrations that are more maintainable than scattered raw SQL strings.

The schema contains four main tables: `users`, `sessions`, `challenges`, and `messages`. The `users` table stores email addresses, password hashes, an optional SSH public key, and timestamps. The `sessions` table associates session IDs with users and expiry timestamps. The `challenges` table stores the mapping between a user and their provisioned container, as well as the learner-visible metadata generated by Gemini, the internal flag, the solution summary, the challenge status, and expiry information. The `messages` table stores tutor conversations keyed to a challenge. This design makes the challenge record the central unit of state: it connects the user, the infrastructure, the evaluation flag, and the tutoring context.

The repository pattern on top of Drizzle is another useful implementation decision. Rather than placing query logic directly in every route handler, the project defines repositories for users, sessions, challenges and messages. This improves readability and reduces duplication. More importantly, it means that behaviours such as “find the user’s active challenge”, “mark a challenge as completed”, or “insert a message pair atomically” are expressed once in the data layer and reused consistently. For a project of this size, this is enough abstraction to improve maintainability without becoming over-engineered.

### 4.5.4 Challenge status model and one-active-challenge enforcement

Challenge lifecycle management became one of the most important implementation concerns. In a platform where each challenge consumes real infrastructure, allowing a user to accumulate multiple active environments would create both resource pressure and interface confusion. The final design therefore enforces a “one active challenge per user” rule through both the application and the database.

At the application level, the launch route checks for an existing active challenge before beginning a new pipeline. If the existing challenge has already expired, the route attempts cleanup and proceeds. Immediately before inserting a new challenge record, it performs a second check to reduce the race window between request start and database write. At the database level, the `challenges` table includes a partial unique index that only applies when `status = 'active'`. This means the storage layer itself rejects attempts to create two active challenges for the same user. The repository catches that constraint violation and translates it into a domain-specific error.

This is a good illustration of how the implemented system matured beyond a minimal prototype. A naïve version of the project could simply have checked “does the user already have a row?” and hoped that the frontend would not send duplicates. The actual implementation anticipates concurrency, delayed expiry, and the fact that challenge provisioning takes long enough for race conditions to matter. It also distinguishes between active, completed, destroyed and expired states instead of overloading a single `completed_at` field with too many meanings. That distinction becomes important in the history view and in user feedback, because an expired challenge is not semantically the same as a successfully completed one.

### 4.5.5 API routes for launch, active state, flag submission, history and destruction

The main challenge lifecycle is expressed through a set of API routes. `POST /api/challenges/launch` initiates the end-to-end provisioning pipeline and streams progress back to the client. `GET /api/challenges/active` returns the currently active challenge, performing lazy expiry cleanup when necessary. `POST /api/challenges/submit-flag` compares the submitted flag with the stored flag for the authenticated user’s challenge and transitions the challenge to completed on success. `DELETE /api/challenges/[id]` marks the challenge as destroyed and initiates container teardown. Additional routes provide challenge history, single-challenge detail retrieval, and tutor message loading.

This route structure is important because it turns the infrastructure pipeline into a learner-facing application flow. The learner does not see “deploy playbook”, “Gemini output”, or “destroy container”; they see launch, connect, ask for help, submit a flag, and review past work. The route handlers therefore form a translation layer between educational actions and technical operations. In that sense, the implementation is not just a backend for infrastructure. It is the mechanism by which the project’s educational model is expressed in software.

## 4.6 Frontend dashboard and user interaction design

### 4.6.1 Dashboard-centric interaction model

The frontend was implemented as a dashboard-oriented interface rather than as a sequence of disconnected forms. This decision supports the project’s goal of making the learner workflow clear: select a challenge, launch it, connect to it, interact with the tutor, submit the flag, and later revisit the result through history. The main dashboard page is responsible for two very different states. When no challenge is active, it presents the vulnerability categories and difficulty selector. When a challenge is active, it pivots to a two-column layout containing the challenge panel and the tutor panel. This prevents the learner from treating challenge launch, exploitation and assistance as unrelated screens.

The categories shown in the dashboard are intentionally aligned with the generation system: they are human-readable representations of the same machine-readable category IDs used in the prompt layer. This helps keep the UI and the backend consistent. Difficulty selection is also simple and explicit, using the three levels defined in the schema. The launch button is only enabled once the user has chosen both a category and difficulty, and the dashboard also warns users if they have not yet uploaded an SSH public key. These are straightforward interface choices, but they reduce avoidable failure paths in the provisioning flow.

### 4.6.2 Streaming launch progress

Provisioning a challenge is not an instant operation. It can involve LLM generation, package installation, file injection, and service restarts, all of which may take tens of seconds. A traditional synchronous request would leave the user with little feedback beyond a loading spinner, which is particularly problematic when debugging or demonstrating the platform. The implementation therefore uses server-sent events to stream launch progress from the server to the dashboard.

The launch route emits progress events for the major phases of challenge creation, including generation, container deployment, vulnerability injection and finalisation. On the client side, the dashboard reads the event stream, updates a progress component, and replaces the category-selection interface with a clear launch status view. This feature is not just cosmetic. It provides evidence that the application layer is genuinely orchestrating a multi-step workflow, and it gives the learner better visibility into where time is being spent. It also made the platform easier to test during development because failures could be localised to a specific stage.

### 4.6.3 Challenge panel, flag workflow and history

When a challenge is active, the left-hand challenge panel presents the key details needed to work on it: title, description, attack vector label, connection details, a hint toggle, countdown timer, flag submission field and a destroy button. The countdown reflects the challenge’s TTL, making the ephemeral nature of the environment visible to the learner. The connection details emphasise the practical, hands-on nature of the system by presenting a real SSH command rather than an abstract “start lab” button.

The flag workflow is designed to close the challenge loop cleanly. Rather than simply recording progress invisibly in the backend, the interface gives immediate feedback on whether a flag is correct or incorrect. The design also clears stale result banners when the user edits the input, which is a small but important usability refinement. Once a challenge is no longer active, it becomes available through the history interface. The history page lists past challenges, including status and message counts, while the detail page allows the learner to review the challenge metadata and the conversation with the tutor. For completed challenges, the solution summary can also be revealed for reflection.

This history functionality strengthens the educational value of the platform. Without it, the tutor and challenge state would vanish once the container expired or was destroyed. By persisting conversations and challenge metadata, the system allows the user to revisit not just what they solved but how they reasoned through it. That makes the platform more than a disposable lab launcher; it becomes a record of practice.

### 4.6.4 Settings and SSH key management

The settings page provides two account-management features that are important to the learner workflow: SSH public key storage and password change. The SSH key feature is particularly central because challenge access depends on it. The user can paste a supported public key into the settings form, and the server validates the key format before storing it. This avoids letting arbitrary malformed text reach the infrastructure layer. The page also allows the user to change their password, which is a simple but necessary account feature in any platform that expects repeat use.

The implementation of SSH key management is another place where the system’s educational requirements and security requirements intersect. A project could simplify matters by using a shared password inside every container, but that would weaken isolation and make it harder to attribute access correctly. By tying challenge access to the stored public key, the implementation keeps the learner experience realistic while preserving a clearer ownership model.

## 4.7 AI tutor implementation

### 4.7.1 Challenge-aware tutoring model

The tutor subsystem addresses the fourth objective in the initial plan: providing guidance without directly giving away the solution. In the final system, the tutor is not a generic chatbot attached to the site. It is challenge-aware and only available when the user has an active challenge or when they are reviewing a past challenge. The tutor route loads the active challenge for the authenticated user, constructs a system prompt that includes the challenge title, category, difficulty, description, attack vector, hint and internal solution summary, and then streams the model’s response back to the client.

This design is important because it means the tutor is grounded in the exact challenge the learner is attempting, rather than in a broad category description. The model can therefore explain methodology and concepts relevant to the specific scenario without having to infer too much from the user’s messages. At the same time, the challenge-aware prompt lets the platform control what kind of help is appropriate. The tutor is explicitly instructed to provide progressive hints, to avoid revealing the full exploit path, and never to output the flag. In pedagogical terms, the tutor is designed to function more like a guided teaching assistant than like a solution oracle.

### 4.7.2 Streaming responses and persistent conversation history

The chat API streams tutor responses using server-sent events. On the frontend, the chat panel appends the user message immediately, creates a placeholder assistant message, and progressively fills it as chunks arrive. This gives the tutor interaction a more responsive feel and aligns with the overall dashboard design, which already uses streaming for challenge launch. The chat panel also supports multiline input, stopping a generation in progress, and a simple markdown-like rendering of code blocks, lists and headings. These details make the tutor more usable for technical explanation, where learners may need commands or examples formatted clearly.

Unlike an earlier design where chat lived purely in component state, the current implementation persists tutor conversations in the database. Messages are stored against a challenge ID, and a separate endpoint loads them when the chat panel mounts. This persistence allows the same conversation to be reviewed later from the history page, which strengthens continuity and supports reflective learning. The message repository also inserts user and assistant responses atomically after a successful stream, avoiding partial or inconsistent records.

### 4.7.3 Tutor guardrails and output filtering

A prompt alone is not a sufficient security control for a tutor that has access to the solution summary. The implementation therefore adds explicit output checks. The chat route buffers the assistant response before releasing it, checks for forbidden content such as `FLAG{...}` patterns, direct reproduction of the stored challenge flag, or verbatim inclusion of the solution summary, and substitutes a refusal message if necessary. This is a heuristic safeguard rather than a formal proof, but it materially improves the system over simply trusting the model to obey the system prompt.

This is an important implementation lesson from the project. Because the tutor is intentionally given privileged context to be useful, the platform must assume that prompt leakage is possible and build a second layer of defence. The output filter acknowledges that a helpful tutoring system can easily become an answer-exfiltration channel if it is not designed carefully. In that sense, the tutor is a microcosm of the whole project: useful AI features must be integrated through controlled interfaces, not through blind trust.

## 4.8 Security, robustness and engineering trade-offs

Several implementation choices were driven by security and reliability concerns rather than by raw feature development. The most visible example is the non-root student model. The learner connects as a `student` user without sudo access, and the prompts instruct Gemini to place the flag somewhere that user cannot read directly. This preserves the distinction between “having access to the machine” and “solving the challenge”. It also reduces the educational temptation to make every challenge trivial by placing the flag in an obvious file. The platform is therefore designed so that exploitation remains necessary even after the learner successfully logs in.

Input validation is another recurring theme. Challenge launch input is constrained to known categories and difficulty values. Chat messages have length limits. Signup and login inputs are validated. SSH keys are checked against an explicit regular expression and length bounds. These checks matter because the platform’s backend can eventually reach the hypervisor, filesystem and LLM. A system like this needs hard boundaries on what counts as acceptable input, otherwise operational problems can quickly become security problems.

The implementation also had to address practical debugging and failure handling. Ansible failures are parsed and surfaced in a human-readable way. Setup command failures are reported with stdout and stderr excerpts. LLM errors are inspected for blocked prompts, empty responses and retryable overload conditions. Containers are destroyed when injection fails after deployment, preventing orphaned vulnerable environments from lingering after partial failure. The platform therefore treats rollback and cleanup as first-class concerns. This is particularly important in a dissertation project because robustness is itself part of the contribution: a cybersecurity training platform that frequently leaves behind stale machines or unexplained errors would not be credible, even if the “happy path” demo worked once.

There were also deliberate trade-offs. SQLite is simpler than a client-server database but less suited to large-scale concurrency; for a one-semester single-project platform this is an acceptable compromise. Tailscale-based networking offers better containment than public exposure but introduces client-side setup friction. The model-based tutor is valuable for guidance but requires layered safeguards to avoid leakage. Containers are more resource-efficient than full virtual machines, but they also require more careful prompt constraints because some vulnerability ideas are inappropriate for unprivileged LXC. These are not shortcomings of the implementation so much as examples of engineering balance. The project consistently prioritised a reliable, demonstrable, and safe architecture over the pursuit of maximum scope.

Several of these decisions can be read as direct responses to the feasibility section of the initial plan. The plan anticipated that running a capable local model might be constrained by hardware and memory availability, so the final implementation used a cloud-hosted Gemini API instead of depending on a large local model. The plan also identified the need for strict access control, escape prevention and a time-based kill switch; in the implemented system these concerns became the non-root `student` account model, Tailscale-restricted routing, unprivileged LXC containers, one-hour challenge TTLs, and the host-side reaper script. In other words, the final architecture did not simply add extra engineering polish after the fact; it operationalised the risks and mitigations that were already identified at planning stage.

## 4.9 Chapter summary

The implementation of VulnForge turned the project’s initial plan into a functioning end-to-end platform for AI-assisted cybersecurity training. The completed system provisions unprivileged Ubuntu containers on Proxmox, configures access through per-user SSH keys, generates challenge definitions with a schema-constrained LLM pipeline, injects those definitions through Ansible, manages challenge state through a typed persistence layer, presents the learner with a dashboard-oriented interface, verifies successful exploitation with server-generated flags, and supports reflective learning through a challenge-aware tutor and history view.

A central theme of the implementation is that each project objective was realised through bounded interfaces rather than through ad hoc glue code. The LLM is constrained by schema and prompt rules. The infrastructure is mediated through Ansible playbooks rather than directly embedded into the frontend. User-visible state is tied to database records rather than transient browser state. Cleanup is handled through both application logic and host-level automation. These choices make the platform more robust and more defensible as a technical artefact.

From a dissertation perspective, the implementation demonstrates that the project is not simply “a website that calls an AI model”. Its main contribution is the integration of several complex concerns—generation, orchestration, isolation, lifecycle management, tutoring, and learner interaction—into a single coherent workflow. The next chapter evaluates how well this implementation satisfied the project aims in practice, including the reliability of generation, the usefulness of the tutor, and the overall viability of generative AI as a mechanism for producing reusable educational security labs.
